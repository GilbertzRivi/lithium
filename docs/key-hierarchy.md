# Key catalog and hierarchy

One combined view of every key and secret in Lithium: where they 
come from, where they live, how long they last, and what they 
protect. The derivation details are in 
[crypto-protocol.md](protocol/crypto-protocol.md); the security 
consequences are in [security-model.md](security-model.md). All 
labels are pinned by tests (`registry_values_are_pinned`).

## Dependency tree

```
data_password
  |-- Argon2id(., salt from mk.enc) -> MK read key -> MK -> KEK = HKDF(MK, salt, "kek/v1") -> DEK(.keyf) -> private keys / secrets
  |-- Argon2id(., root.salt) -> password_root --\
server (OPAQUE login) -> server_dek -------------+--> HKDF(server_dek, salt=password_root, ".../combined/v1")
                                                      -> combined_root -> db_dek -> local SQLite
  (server_dek is wrapped under the OPAQUE export_key, AAD "lithium/dek-wrap/v1")

server MK
  |-- KEK -> DEK(server .keyf) -> server identity keys
  |-- derive_secret32 -> server db_dek ("lithium/db-dek/v1") -> users fields
  |-- derive_secret32 -> jwt-secret ("lithium/jwt-secret/v1")
msg_key (random per message, RAM only, TTL 24 h) -> message content in the server DB
```

The key property: the client's `db_dek` needs **both** 
`password_root` (from the data password) and `server_dek` (from 
the server) at the same time. Losing either factor makes the local 
database impossible to decrypt, that's the deliberate two-factor.

## Client: at-rest data encryption

| Key | Type | Derivation / source | Storage | Lifetime / rotation | Protects |
|-----|------|--------------------|---------|--------------------|----------|
| Master Key (MK) | 32 B random | wrapped `AES-256-GCM-SIV(MK, Argon2id(data_password, salt), aad="lithium/mkfile/v1")` | `keystore/user/mk.enc` | rotated every 1 h (`MkRotator`, 30 s tick) | all `.keyf` files (through the KEK) |
| KEK | 32 B | `HKDF(MK, file_salt, info="kek/v1")` | not stored (derived on use) | with the MK | wrapping the DEK in a `.keyf` |
| `.keyf` DEK | 32 B random | random per file | in the `.keyf`, wrapped under the KEK | rewrapped on MK rotation (value unchanged) | the key/secret payload in the file |
| root.salt | 32 B random | random per install | `keystore/user/root.salt` | fixed | salt for `password_root` |
| password_root | 32 B | `Argon2id(data_password, root.salt)` (64 MiB, t=3, p=1) | RAM only (cache) | session (until `lock_keystore`) | input to `combined_root` |
| server_dek | 32 B random | random at `register`; wrapped under the OPAQUE `export_key` (AAD `"lithium/dek-wrap/v1"`) | **on the server**; returned at login; never on the client disk | until account deletion | input to `combined_root` |
| combined_root | 32 B | `HKDF(server_dek, salt=password_root, info="lithium/user-provider/combined/v1")` | RAM only | session | source of `db_dek` |
| db_dek (client) | 32 B | `HKDF(combined_root, info="lithium/db-dek/v1")` | RAM only | session | the `*_enc` fields in the local SQLite |
| export_key (OPAQUE) | OPAQUE output | derived client-side at register/login | RAM only | per operation | wraps `server_dek` |

## Client: per-contact keys

Stored in `self_state` / `peer_state`, encrypted under `db_dek` 
(AAD `lithiumd/contact-self/v1` / `lithiumd/contact-peer/v1`). Each 
contact has an independent set, compromising one doesn't touch the 
others.

| Key | Type | Source | Lifetime / rotation | Role |
|-----|------|--------|--------------------|------|
| E2E encryption | X25519 + ML-KEM-1024 | random at `create_invite` | persistent per contact | KyberBox of the E2E content |
| E2E signatures | Ed25519 + ML-DSA-87 | random at `create_invite` | persistent per contact | dual signature of E2E messages |
| Mailbox keys | 3x X25519 (in / out_cur / out_next) | random at `create_invite` | `out` rotated every 32 sends | mailbox addressing |
| Bootstrap | X25519 + ML-KEM (from the `lci1:` code) | from the invite | removed after peer ack + ratchet | the first message to a contact |
| RX keyring (reply) | X25519 + ML-KEM per send | fresh per send, numbered by `seq` | a window of 32 from `ack_seq`; older ones deleted (zeroized) | reply-key ratchet |
| Prekeys | 5x X25519 + ML-KEM | generated at the first send; private in the `prekeys` table (AAD `lithiumd/prekey/v1`) | removed after use (`take_prekey`) | recovery after desync |

## Transport (client and server)

| Key | Type | Source | Storage / TTL | Role |
|-----|------|--------|---------------|------|
| Server identity (long-term) | X25519 + ML-KEM (enc) + Ed25519 + ML-DSA (sign) | the server's `KeyManager`; the public part pinned on the client in `server.identity` | persistent | Shake, signing the server's responses |
| Transport session keys | X25519 + ML-KEM ephemeral | generated by the server after each response | `EphemeralStore`; TTL 60 s (Shake) / 120 s (Session) | encrypting subsequent requests |
| Ephemeral client keys | X25519 + ML-KEM + Ed25519 + ML-DSA | per request (the `KeysInHeaders` endpoints) | only for the request's duration | anonymous transport (shake / send / fetch / revoke) |

## Server

| Key | Type | Derivation / source | Storage | Rotation | Protects |
|-----|------|--------------------|---------|----------|----------|
| Server MK | 32 B | `TpmMkProvider` (sealed KEYEDHASH) or `PlainFileMkProvider` | TPM blob (`LITHIUM_TPM_SEALED_PATH`) or file | every 1 h | the server's `.keyf` files |
| Server db_dek | 32 B | `derive_secret32("lithium/db-dek/v1")` from the server MK | RAM only | with the MK (rewrap) | the `users` table fields |
| msg_key (per message) | 32 B random | random at `add_message` | `EphemeralStore`, TTL 24 h | one-time (`store.take` on fetch) | message content on the server (AAD `message-content/v1` || mailbox) |
| JWT secret | 32 B | `derive_secret32("lithium/jwt-secret/v1")` | RAM only | regenerated on MK rotation | HS256 signing of JWT tokens |
| OPAQUE ServerSetup | OPAQUE setup | generated; sealed (label `lithium/opaque-server-setup/v1`) | sealed blob | persistent | OPAQUE authentication of all accounts |
| TPM sealing parent | ECC P-256 (restricted decryption) | deterministically from the TPM owner seed | **never persisted** | - | the parent that seals the Server MK |

The client and server `db_dek` use the same label 
`"lithium/db-dek/v1"`, but come out of **different** MK providers 
(client: `combined_root`; server: the server MK), they are two 
different keys.

## Leak analysis

What compromising a given element exposes:

| Compromise | What's exposed | What's still protected |
|------------|----------------|------------------------|
| The client disk alone (no password) | nothing, `.keyf` and SQLite are encrypted; MK behind Argon2 | everything |
| Client disk + data password, **without** the server | the MK and `.keyf` keys are recoverable | the local database (no `server_dek` means no `db_dek`), the deliberate two-factor |
| Disk + password + an active server session | this user's full data | (the model assumes this is the user themselves) |
| `db_dek` leaking from RAM | the local database while the key lives | the MK, the `.keyf` keys |
| Relay server takeover | encrypted `users` fields and encrypted mailboxes | E2E content (per-contact keys only on the clients); a restart destroys `msg_key`, so pending messages are permanently undecryptable |
| One contact's keys | that one contact | the other contacts (isolation) |
| Breaking ML-KEM **or** X25519 on its own | nothing, the other half of the hybrid still protects | content (both must break) |
