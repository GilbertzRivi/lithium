# Lithium

**A post-quantum, end-to-end encrypted messenger designed for high-security environments.**

Lithium is not a consumer messenger. It was built for organizations and users who cannot allow the contents of their communications to be accessible to anyone other than the direct participants — including the operator, infrastructure provider, or a court issuing a disclosure order.

> **Design priority:** Content confidentiality takes precedence over convenience. Whenever the two conflict, Lithium chooses confidentiality.

---

## Who Lithium is for

Lithium is intended for environments where:

- the server, operator, or infrastructure may be **monitored, compromised, or legally compelled to cooperate**,
- conventional messengers — even encrypted ones — are unacceptable because they require **trust in the operator**,
- an organization needs a messenger that **mathematically prevents the service provider from disclosing message content**,
- there is a realistic risk that a **client device's disk may be seized** by an adversary,
- regulatory or operational requirements demand **minimal data retention** and no ability for the operator to reconstruct message history.

Example target users include law firms, companies involved in negotiations and mergers, journalistic organizations and NGOs operating in hostile environments, and financial institutions that require confidential internal communication.

---

## Key properties

### The operator is mathematically unable to disclose message content

The Lithium server is treated as a hostile relay. It stores and forwards encrypted data only. It has no access to:

- message content,
- users' private keys,
- relationships between correspondents, because mailbox addresses are cryptographically pseudorandom.

Even under legal compulsion, the operator cannot provide plaintext — not because it refuses to do so, but because **it does not possess it**.

### Post-quantum resistance

All cryptographic operations use a hybrid construction: a classical algorithm and a post-quantum algorithm are used in parallel. Breaking either one alone does not compromise the system — both must be broken.

| Purpose                 | Algorithms                          |
|-------------------------|-------------------------------------|
| Key exchange            | X25519 + ML-KEM-1024 (NIST PQC)     |
| Symmetric encryption    | AES-256-GCM-SIV                     |
| Digital signatures      | Ed25519 + ML-DSA-87 (NIST PQC)      |
| Key derivation          | HKDF-SHA256                         |
| Password hashing        | Argon2id                            |

The post-quantum algorithms ML-KEM-1024 and ML-DSA-87 were standardized by NIST in 2024 for deployments requiring resistance to quantum-computer attacks.

### Forward secrecy — past messages remain secure after key compromise

- **Per message:** every message contains fresh ephemeral keys. Compromising the current key does not allow previous messages to be decrypted.
- **Per generation:** mailbox keys rotate every 32 messages, and old private keys are securely erased.
- **Transport:** transport-session keys have a TTL of 60–120 seconds. Compromising a session does not allow earlier traffic to be decrypted.

### Two-factor protection of local data

Decrypting data stored on the device requires both:

1. the user's password, protecting the local data,
2. a server-side component, with the DEK retrieved at every login.

The DEK, or Data Encryption Key, is generated randomly by the client during registration. The server does not create it, know it, or have the ability to reconstruct it. The client sends it to the server only after encrypting it with the user's password. The server stores it as an opaque blob and returns it at each login. In this design, the server acts solely as storage and cannot use the DEK without the client's password.

Seizing the device's disk without knowing the password **and** having access to the server yields no plaintext. This is a design decision, not a limitation.

### Cryptographic uniqueness per installation

Every daemon installation independently generates its own cryptographic material — asymmetric keys, a master-key seed, and mailbox keys — using the operating system's cryptographically secure random number generator (CSPRNG). There is no shared secret or installation-wide seed. Two installations on two devices have no cryptographic relationship, even if they belong to the same user.

### Server identity pinning and substitution protection

The server identity — a set of four public keys: X25519, ML-KEM-1024, Ed25519, and ML-DSA-87 — is stored in a binary `server.identity` file generated by the server on first startup. The client daemon loads this file and uses it to verify every server response.

`server.identity` has no URL or endpoint it can be fetched from automatically, and never will — this is a deliberate, separate security layer, independent of the relay server's URL (which is set through its own command, `set_server_url`, and is only used to open an HTTP connection, never to verify the server's identity). The file must be delivered over an out-of-band channel and uploaded manually, by a conscious user action. This is intentional: if the client could pull `server.identity` over the network on its own, an attacker who had taken over the server could swap its keys without users ever noticing — automatic retrieval of a new identity would defeat the entire protection this file is meant to provide.

As a consequence, **any change to the server keys — whether caused by substitution or external interference — immediately and permanently breaks communication with all existing clients.** A client cannot connect to a server whose identity it does not recognize. Restoring communication requires a deliberate user action: manually installing a new `server.identity` file. This is intentional — silently replacing server keys without the users' knowledge is impossible.

### One-time messages

Messages are atomically deleted from the server on first retrieval. The server stores no message history. History exists only in the client's local database, encrypted independently on each device.

### Identity verification without trusting the server

The server is not a trust anchor. A correspondent's identity is verified by comparing an emoji fingerprint over an out-of-band channel, such as a phone call. The server cannot impersonate either party.

## System architecture

```text
┌─────────────────────────────────────┐
│  lithiumg  (GUI — Linux / Windows)  │
│  User interface                     │
└────────────────┬────────────────────┘
                 │ JSON / Unix socket / Windows named pipe
                 │ (local connections only)
┌────────────────▼────────────────────┐
│  lithiumd  (client daemon)          │
│  Private keys · SQLite · crypto     │
│  The only component with plaintext │
└────────────────┬────────────────────┘
                 │ HTTPS
                 │ KyberBox (X25519 + ML-KEM-1024)
                 │ dual-sign (Ed25519 + ML-DSA-87)
┌────────────────▼────────────────────┐
│  lithiums  (relay server)           │
│  PostgreSQL · ciphertext only       │
│  Never sees plaintext               │
└─────────────────────────────────────┘
```

The system consists of four components:

| Component      | Role                                                                                   |
|----------------|----------------------------------------------------------------------------------------|
| `lithium_core` | Cryptographic library shared by the daemon and server                                  |
| `lithiumd`     | Client daemon that stores keys, performs encryption, and exposes IPC to the GUI        |
| `lithiumg`     | Graphical interface that communicates with the daemon and never accesses key material  |
| `lithiums`     | Relay server that accepts and forwards encrypted messages using PostgreSQL             |

### Cryptographic isolation

Private keys and plaintext exist only inside `lithiumd` on the user's device. The GUI, `lithiumg`, communicates with the daemon over a local socket and never gains access to key material. The server, `lithiums`, sees only encrypted blobs and does not participate in any end-to-end cryptographic operation.

## Message flow

### Sending

```text
The user enters text in the GUI
  → the GUI sends IPC to the daemon: contact_send(contact_id, plaintext)
  → the daemon encrypts it using WireV1
    (X25519 + ML-KEM + AES-256-GCM-SIV + dual-sign)
  → the daemon sends the encrypted blob to the server over HTTPS
  → the server wraps the blob with an additional random per-message key
  → the server stores it in PostgreSQL with a 24-hour TTL
```

### Receiving

```text
The user clicks Fetch in the GUI
  → the GUI sends IPC to the daemon: contact_fetch(contact_id)
  → the daemon derives the mailbox address using ECDH with mailbox keys
  → the daemon retrieves blobs from the server over HTTPS
  → the server atomically returns and deletes the messages from the database
  → the daemon decrypts the messages and verifies the signatures
  → the daemon stores plaintext in the local SQLite database
  → the GUI displays the history
```

### Adding a contact through invitation exchange

Pairing is performed by exchanging invitation codes, `lci1:HEX`, outside the server over an out-of-band channel such as email, phone, or another messenger. An invitation code contains public keys only — no private data.

```text
Party A: [New contact] → lci1:HEX code containing A's public keys
Party B: selects that contact, pastes A's code into the invite field + [Accept invite]
        → contact B now has peer_set=true, and gets its own lci1:HEX code with B's public keys
Party A: selects that same (still-pending) contact, pastes B's code into the same field + [Accept invite]
→ both parties now have peer_set=true → messaging is enabled
```

Both sides use the same "Accept invite" button — what differs is only which contact is currently selected (a fresh one with no `contact_id` yet, versus the one A created earlier that's still waiting for a reply). The separate "Reply to invite" button in the GUI does not consume a pasted code at all — it just regenerates/redisplays A's own `lci1:HEX` code for the selected pending contact, useful if it needs to be resent.

After the exchange, both parties verify the emoji fingerprint by voice or in person. Only this step confirms that no man-in-the-middle attack occurred during invitation exchange.

## Security properties — overview

| Property                              | Mechanism                                                                                                                |
|---------------------------------------|--------------------------------------------------------------------------------------------------------------------------|
| Post-quantum resistance               | ML-KEM-1024 + ML-DSA-87 in parallel with X25519 + Ed25519; both algorithm families must be broken                        |
| Per-message forward secrecy           | Fresh ephemeral X25519 + ML-KEM keys in every message using a ratchet; old keys are erased after acknowledgement         |
| Per-generation forward secrecy        | Mailbox keys rotate every 32 messages; the sender's old private keys are erased                                          |
| Transport forward secrecy             | Session-key TTL of 60–120 seconds; ephemeral X25519 + ML-KEM keys per request in Shake mode                              |
| Post-compromise security              | Mailbox-key rotation and prekey recovery allow security to be restored after state compromise                            |
| No plaintext on the server            | Content is encrypted by the client before reaching the server; the server adds a second layer without reading it         |
| One-time messages                     | Atomic deletion on first retrieval; the server cannot reconstruct them                                                    |
| Ephemeral message keys                | Per-message keys exist only in server memory; restarting the server destroys them                                        |
| Handle-enumeration protection         | Attempting to register an existing login returns success, producing no distinguishable response                          |
| Anti-replay                           | `SHA256(body)` is stored for 600 seconds; request timestamps are validated within ±60 seconds                            |
| One-time JWT                          | A token is consumed when used, so a captured token cannot be replayed                                                     |
| Database-field isolation              | Every field is encrypted under a separate AAD domain; incorrect AAD causes decryption to fail                            |
| Size padding                          | Bodies are padded to 32–64 KB blocks and headers to 4–8 KB, hiding message length and operation type                     |
| Identity verification                 | Out-of-band emoji fingerprint comparison makes MITM during invitation exchange detectable by users                      |
| Two-factor local-data protection      | Password + server component; a seized disk without the password and server access yields no data                         |
| Memory zeroization                    | All secret types erase memory when dropped using `zeroize`; keys are not left behind in memory                           |
| Atomic file operations                | Keys are written using `tmp + rename + fsync`; interruption does not corrupt state                                       |
| Crash-safe key rotation               | An incomplete rotation is detected and completed at startup                                                              |
| Server identity pinning               | The client verifies every response against `server.identity`; changing server keys disconnects all existing clients      |
| Emergency account removal             | Registration creates a one-time 32-byte capability; only its SHA-256 is stored in the database; it can delete the account without login after device loss |
| Per-installation uniqueness           | All keys and seeds are generated independently using the device CSPRNG; no installation-wide shared secrets              |
| Client-generated DEK                  | The client creates the Data Encryption Key randomly and sends it password-encrypted; the server stores an opaque blob     |

## What Lithium deliberately is not

The following limitations are **design features**, not defects. They follow directly from the security model.

- **It is not a mass-market messenger.** There are no groups, channels, presence indicators, reactions, threads, or avatars.
- **There is no password recovery.** Losing the keystore password means permanent loss of access. No reset mechanism exists — not by email, SMS, or through the operator. This is intentional.
- **It stores no server-side history.** Messages are deleted when retrieved. History exists locally only.
- **It does not support multiple devices.** One account equals one daemon on one device, with no cross-device synchronization.
- **It has no push notifications.** It uses a pull model in which the client polls the server. There is no APNs, FCM, or other push infrastructure.
- **It does not guarantee delivery of every message.** The server may refuse service, lose data, or affect availability. The server is not a trusted component, and that carries an operational cost.
- **It does not operate fully offline.** Decrypting local data requires a component retrieved from the server. This is intentional: loss of access is preferred over the risk of data disclosure after device theft.
- **It is not interoperable.** It uses its own WireV1 protocol and invitation format, deliberately without compatibility with Signal, Matrix, XMPP, or other systems.
- **There is no web version or SaaS deployment.** A locally running daemon is required. The operator cannot provide conventional SaaS availability guarantees without weakening the trust model.

## Deployment

### Requirements

- **Rust**, stable toolchain, 2024 edition, for building from source
- **PostgreSQL** for the `lithiums` relay server
- **SQLite**, embedded in the `lithiumd` client daemon
- **Linux or Windows** for both client and server
- **Linux: `libgtk-3-dev` and `libappindicator3-dev`** (or the `libayatana-appindicator` equivalent) — `lithiumd` embeds a system tray icon; without these packages the build fails at the pkg-config step for the `*-sys` crates

### Building

```bash
# All components
cargo build --release

# Relay server only
cargo build --release -p lithiums

# Client only: daemon + GUI
cargo build --release -p lithiumd -p lithiumg
```

### Running the relay server

`lithiums` listens over plain HTTP. TLS is terminated by a reverse proxy such as nginx or Caddy in front of the server process.

The target deployment environment is Docker Compose. All server configuration is supplied through environment variables, the database password is provided through a file such as a Docker secret, and the key directory is mounted as a volume.

```bash
export DB_HOST=localhost
export DB_USER=lithium
export DB_PASSWORD_FILE=/run/secrets/db_password
export DB_NAME=lithium
export LITHIUM_KEYS_DIR=/var/lib/lithiums/keys
export LITHIUM_BIND=0.0.0.0
export LITHIUM_PORT=4108

lithiums
```

On first startup, the server generates its own keys in `LITHIUM_KEYS_DIR` and writes a `server.identity` file containing four public keys — X25519, ML-KEM-1024, Ed25519, and ML-DSA-87 — in a binary format with magic bytes. This file is the sole artifact used to distribute the server identity and must be delivered to users over an out-of-band channel.

### Configuring the client daemon

```bash
export LITHIUMD_SERVER_IDENTITY=/path/to/server.identity  # optional; default: {data_dir}/server.identity

lithiumd
```

The relay server address is **not** an environment variable — it is set after the daemon starts via the IPC command `set_server_url` (in the GUI: during the first-run setup step). The server identity (`server.identity`) is likewise uploaded via IPC (`set_server_identity`), not pointed to by path — `LITHIUMD_SERVER_IDENTITY` only changes where the daemon keeps its local copy once uploaded. The `server.identity` file itself must still be delivered by the server administrator over an out-of-band channel before the first connection. Details: [`docs/ipc-reference.md`](docs/ipc-reference.md#set_server_url).

### Running the GUI

```bash
# The daemon must already be running
lithiumg
```

On first launch, the GUI guides the user through setup, in this order:

1. Enter the relay server's URL.
2. Import the `server.identity` file to verify the server identity.
3. Set a keystore password that encrypts private keys on disk.
4. Enter an account name and server-account password.
5. Register the profile on the server. After registration, the GUI displays the **emergency account-removal capability** described below, which must be saved.
6. Unlock local storage (initializes the local SQLite database — a single click, happens right after registration).

### Emergency remote account removal

During registration, the server generates a random 32-byte token called `remote_delete_capability` and returns it to the client. The database stores only the token's SHA-256 hash, so the server does not know the plaintext capability value.

If a device is lost or stolen, the user can delete the server-side account without logging in. The capability and access to the `server.identity` file are sufficient:

```text
GUI → [Emergency account removal] → paste capability → [Remove]
```

The capability requires neither a password nor an active session. It cannot be reconstructed: losing it permanently removes the owner's ability to delete the account. Administrative intervention is not an alternative. Handles are not stored in plaintext; the database contains only a UUIDv5 derived from the handle and deterministically encrypted using a server key. The operator cannot identify or locate the record using the handle, username, or any other plaintext identifier.

## Master-key rotation

The daemon and server rotate their master keys every hour by default. Rotation is atomic and crash-safe: an incomplete rotation is automatically detected and completed at startup. Rotation rewraps keys under the new master key without re-encrypting the database contents.

## Security model — summary

Lithium assumes that:

- the server is or may become hostile, monitored, or legally compelled to cooperate,
- the client disk may be seized,
- the operator is not, and cannot be, trusted with content confidentiality.

In response to those assumptions:

- the server is mathematically unable to decrypt message content,
- the operator does not participate in user pairing or identity verification,
- compromising the server does not provide access to message history,
- compromising the client disk without the password and server access does not provide access to data,
- losing key material causes loss of access, never recovery by a third party.

**Lithium is not designed to be convenient. It is designed to be difficult to betray.**
